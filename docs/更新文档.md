# 部署流程

```bash
unzip TARS-chat.zip && cd TARS-chat
bash scripts/install.sh        # 一键装依赖 + systemd
python scripts/test_servo.py   # 先单独测舵机会不会动
sudo systemctl start tars-chat # 启动
# 浏览器打开 http://<树莓派IP>:8080
```

V0.1.0.0-20260512
fate:更新几个设计决定:
1. 三种 Provider 用同一个抽象基类:OpenAIProvider、AnthropicProvider、OllamaProvider 都继承 BaseLLMProvider,只用实现 _chat() 一个方法。
换 Provider 只需要改 config.yaml 一行,不用动代码。
2. LLM 返回的是结构化指令,不是纯文本。LLM 输出 JSON {say, expression, motion},然后 DialogueOrchestrator 把三个字段分发给表情、舵机、TTS 三路并发执行——这样机器人会"一边点头一边说话",而不是"先说完再点头",体感差很多。
3. 容错降级:小模型(0.5B 这种)经常不遵守 JSON 格式,做了正则清洗 + JSON 解析失败时降级把整段当作 say,机器人虽然不动作但至少能说话,不会崩。
4. Anthropic API 的特殊处理:它的 system prompt 是单独字段,不在 messages 数组里,这个坑我已经在 AnthropicProvider._chat 里处理了。

几个使用建议:
最快上手:直接填 OpenAI key,5 分钟跑通。把 key 放环境变量别提交到 git
追求隐私:外接一台电脑跑 Ollama (OLLAMA_HOST=0.0.0.0:11434),让 Pi 通过局域网调用,比在 Pi 上本地跑流畅 10 倍
改性格:config.yaml 里改 system_prompt,记得保留 JSON 输出格式说明,否则编排器拿不到表情/动作字段
测试:不接舵机也能验证,mock 模式会把所有动作打日志

详细文档在 docs/LLM.md,三种后端的接入步骤、故障排查都在里面。
